# Prompt for Claude Code — Investigate Why Every Major Takes 5–6 Years to Graduate

> Copy everything below the line into Claude Code as a single message.

---

<role>
You are a senior debugging engineer. There's a regression in Prototype 2: every major's generated schedule spans 5–6 academic years instead of the expected 4. Your job is to **investigate first**, write up the root cause, and only after I approve the findings do you fix anything. Do not modify engine logic, requirement definitions, or course data until the investigation report is approved. Read `CLAUDE.md` at the repo root and skim `Prototype 2 Website - source code/js/engine.js` before forming hypotheses.
</role>

<context>
**Symptom**: schedules generated by Prototype 2 currently span 15–18 quarters (≈ 5–6 years) instead of the expected 12 quarters (4 years × Fall/Winter/Spring). This is universal across majors — not a one-off.

**Why this is suspicious**:
- Most UCSC majors require ≈180 units to graduate. 4 years × 3 quarters × ~17 units/quarter ≈ 204 units of capacity. So 180 units should fit comfortably in 12 quarters.
- After the recent placement-balancing pass, quarter loads should average 15–19. If schedules are spanning 18 quarters, that implies the engine is scheduling **roughly 270–300 units of coursework** — far more than 180.
- The user's intuition: either (a) some courses are being placed twice, or (b) some requirements are being marked unmet even after a course that should satisfy them was placed — causing the selector to keep adding redundant courses.

**The user explicitly does not want destructive fixes during investigation.** If you can't find a smoking gun, they will manually audit the major requirements in `Prototype 2 Website - source code/js/majors.js`. So this prompt prioritizes a high-quality report over speculative patches.
</context>

<goal>
Produce a single file `Prototype 2 Website - source code/INVESTIGATION_REPORT.md` with:
1. A clear ranked list of root-cause candidates, each scored Confirmed / Strongly Suspected / Possible / Ruled Out.
2. Evidence for each (line numbers, sample course lists, counts, diffs).
3. A recommended fix scope — minimum changes to bring schedules back to 4 years without touching anything that's working.
4. A list of things you suspect but couldn't confirm without my input (e.g. "I think CSGD_BS's UD_ELECTIVE category is misconfigured — please verify against the source PDF").
</goal>

<deliverables>
Run these phases sequentially. Use `TodoWrite`. **Stop after Phase 4 and show me the report before fixing anything.**

### Phase 1 — Reproduce the bug
1. From `Prototype 2 Website - source code/`, run `node test_smoke.js` and capture the full output.
2. For CS_BA + AI/ML concentration specifically, instrument a one-off Node script (`scripts/inspect_schedule.js`) that:
   - Loads the engine the same way `test_smoke.js` does.
   - Generates a schedule with a fresh-freshman profile.
   - Prints the **entire placed schedule** quarter-by-quarter with course code, title, units, courseType, and the requirement category it was selected to satisfy.
   - Prints the **selected[] list before placement** so we can see if the problem is selection or placement.
   - Prints a **duplicates report**: any course code appearing more than once in the placed schedule.

   - Prints **total units placed** and compares against `major.totalUnitsRequired`.
3. Save the dump to `scripts/output/cs_ba_ai_ml_dump.txt` so we can reference it.

### Phase 2 — Test the leading hypotheses against the dump
For each hypothesis below, do the specific check listed. Mark each Confirmed / Strongly Suspected / Possible / Ruled Out with evidence.

**H1 — Duplicate placement.** Does any course code appear in the placed schedule more than once? Look for `dedupe` / `Set` usage in `placeIntoQuarters` and the selector.

**H2 — `pick_one` not detecting completed equivalents.** Does the engine schedule both alternatives of a `pick_one` category (e.g. AM 10 *and* MATH 21)? Grep `selectCourses` / `walk` for how it skips `pick_one` when an alternative is already in `selected[]`.

**H3 — `choose_group` adding all groups.** Does the engine schedule courses from multiple groups of a `choose_group` category instead of just one? Look at how the chosen group is locked in.

**H4 — `pick_n` over-picking.** For each category of type `pick_n`, count placed courses satisfying it. Does count exceed `n`? Look for off-by-one in the slice/take logic.

**H5 — `expandPrereqs` cascade.** How many courses does the prereq-expansion step add? Print the diff between `selected[]` before and after `expandPrereqs`. Is anything added that's already in the schedule? Is anything added whose prereqs themselves trigger more additions in an unbounded way? `CLAUDE.md` says it's capped at 4 levels — confirm the cap is still enforced.

**H6 — GE auto-satisfied not respected.** `GE_REQUIREMENTS` in `data.js` has an `autoSatisfiedBy` field (e.g. MF auto-satisfied by CSE 16, DC by CSE 115A). Is the engine still picking a separate GE course even when the major already covers it? Trace `pickGECourses` (or its Prototype 2 equivalent) and confirm it checks `autoSatisfiedBy` against the major selections.

**H7 — `completedCourses` not respected.** Set up a profile where the student has already completed 3 lower-div courses. Does the engine schedule them again? Look for the dedup-against-completed step in selection.

**H8 — Concentration-driven over-selection.** The concentration logic was added to drive `pick_n` rankings. Could it be **additively** selecting concentration courses on top of the required `pick_n`? E.g. if BREADTH needs 3 and we have 5 concentration-tagged courses, does the engine take 3 or 5?

**H9 — Filler pool over-adding.** The new placement code adds filler courses when a quarter is below 15 units. Does it cap total fillers? Could it keep adding fillers in a loop that pushes the schedule past 12 quarters? Check whether fillers count against `totalUnitsRequired` or are extra.

**H10 — Cross-listed courses counted twice.** With the refreshed course DB, some courses have a `crossListed` field. Could the validator see "AM 10" and "MATH 21" as separate even though one satisfies both? Could the selector be adding both?

**H11 — Validator mismatch causing rescheduling.** For each "met" requirement in the validator output, identify the course that satisfied it, then check whether the selector still tried to schedule additional courses for the same requirement. This is the user's hypothesis: "requirements covered by some class but didn't count." 

**H12 — Wrong major definition.** Compare `Prototype 2 Website - source code/js/majors.js` against `Prototype 1 Website - source code/js/majors.js`. Were any categories accidentally duplicated, mis-typed, or given inflated `n` values during the Prototype 2 setup? Run `diff` between the two and show me anything non-trivial.

**H13 — `totalUnitsRequired` over-padding.** Does the engine try to pad to `totalUnitsRequired` even after all categorical requirements are met? If `totalUnitsRequired` is wrong (e.g. accidentally set to 220 instead of 180 for some major), every schedule would be inflated by ~40 units.
